§ 01

Node.js & Express — depth check

5 questions · candidate claims 3 yrs
Q 1.1EASY
In which phase of the Node.js event loop are setImmediate() callbacks executed?
A.Timers phase
B.Pending callbacks phase
C.Check phase
D.Poll phase
Expected Answer
C. Check phase. The event loop has six phases — timers, pending callbacks, idle/prepare, poll, check (where setImmediate callbacks run), and close callbacks. A senior should also note that setImmediate fires after I/O events in the poll phase, which is its main practical difference from setTimeout(fn, 0).
Follow-up
  • Where does process.nextTick sit in this picture? (Answer: microtask queue, runs between every phase transition)
Q 1.2MEDIUM
What does this Express middleware print when a GET request hits /test?
app.use((req, res, next) => {
  console.log('A');
  next();
  console.log('D');
});
app.use((req, res, next) => {
  console.log('B');
  next();
});
app.get('/test', (req, res) => {
  console.log('C');
  res.send('ok');
});
A.A, B, C
B.A, B, C, D
C.A, D, B, C
D.A, B, D, C
Expected Answer
B. A, B, C, D. The middleware chain unwinds back up after the response handler. Code after next() runs once control returns. This pattern is the foundation of response-timing middleware (e.g., logging duration). A senior should mention that calling next() in an async middleware without awaiting it can break this unwinding.
Q 1.3MEDIUM
Your StriveScan project says you built "high-concurrency event scanning modules." For a Node.js service handling 10k concurrent scan events per minute, which is the right concurrency primitive — cluster module, worker_threads, or just async I/O? Justify briefly.
Expected Answer
A strong answer separates I/O-bound from CPU-bound work:
  • Async I/O alone handles the bulk if scans are I/O-bound (DB writes, API calls) — Node's single-thread event loop is built for this.
  • cluster horizontally scales across CPU cores via process forking; useful when you want to saturate a multi-core machine without container orchestration.
  • worker_threads only matter for CPU-bound transforms (hashing, parsing big payloads, compression) where you'd otherwise block the loop.
Red flag: if candidate jumps to worker_threads immediately for I/O-heavy work, that suggests shallow Node experience.
Q 1.4HARD
In an Express + Node service handling file uploads, you notice memory keeps climbing under load even though uploads complete successfully. Name two likely causes and how you'd diagnose.
Expected Answer
Common causes:
  • Buffering instead of streaming: reading the full file into memory (req.body, multer.memoryStorage) instead of piping to disk/S3.
  • Unclosed streams or event listeners accumulating on long-lived emitters (memory leak via closure references).
  • Large response caching in middleware (e.g., compression buffers).
Diagnosis: --inspect + Chrome DevTools heap snapshots, clinic.js doctor, or node --heap-prof. Bonus if they mention V8's old-generation growth as the telltale sign.
Q 1.5MEDIUM
You're using NestJS (claimed in skills). What is the difference between a Provider, a Module, and a Controller?
Expected Answer
  • Controller: handles incoming requests, routes them — the HTTP layer.
  • Provider: any class with @Injectable() — services, repositories, factories. The DI container resolves these.
  • Module: a logical grouping with @Module({ imports, controllers, providers, exports }). Defines a scope for DI and what it shares with other modules.
Red flag: If candidate can't articulate dependency injection scope or hasn't actually written a NestJS module, the NestJS line in their resume may be over-claimed.

Laravel & PHP — 5-year claim

5 questions · expect deep answers
Q 2.1EASY
What's the practical difference between with() and load() on an Eloquent query?
Expected Answer
with() is eager loading at query time — User::with('posts')->get() runs two queries up front.
load() is lazy eager loading after you already have the collection — $users->load('posts'). Useful when relationships are conditional.
Both solve the N+1 problem. A 5-year Laravel dev should also mention loadMissing, withCount, and constraining eager loads via closures.
Q 2.2MEDIUM
Resume mentions Laravel Passport and Sanctum. When would you pick Sanctum over Passport?
A.Sanctum supports OAuth2 flows; Passport doesn't.
B.Sanctum is lightweight — token-based or SPA cookie auth; Passport is heavier (full OAuth2 server) and only needed when you're authorizing third-party clients.
C.Sanctum issues JWTs; Passport doesn't.
D.They're functionally identical; Sanctum is just newer.
Expected Answer
B. Sanctum = SPAs, mobile apps, simple API tokens (opaque, DB-backed). Passport = full OAuth2 authorization server (authorization code, client credentials, password grants), needed only when you're letting third parties build apps against your API.
Q 2.3MEDIUM
What does the Service Container do in Laravel, and how is it different from a Service Provider?
Expected Answer
Service Container is Laravel's IoC / DI container — resolves classes, manages bindings, performs auto-wiring via reflection.
Service Provider is the configuration entry point — where you bind(), singleton(), register macros, publish config, and bootstrap packages. Providers register things into the container.
A 5-year Laravel dev should also know about register() vs boot() ordering and deferred providers.
Q 2.4HARD
A queued job is failing intermittently in production. Walk me through how you'd debug it — what's logged, what Artisan commands you'd use, and how you'd guarantee idempotency.
Expected Answer
Strong answer covers:
  • failed_jobs table — schema, payload, exception stack trace.
  • php artisan queue:failed, queue:retry {id}, queue:flush.
  • $tries, $backoff, $timeout properties; failed() method on the job class for cleanup.
  • Idempotency: use ShouldBeUnique interface, or guard with a DB lookup / Redis lock keyed on a business ID. Database transactions wrapping side-effects so retries don't double-charge / double-send.
  • Bonus: Horizon dashboard for observability, separating queues by criticality.
Q 2.5EASY
Resume mentions Livewire and Flux (Speed of Thought, 2025). What problem does Livewire solve, and what's the tradeoff vs. Vue/React?
Expected Answer
Livewire lets you build dynamic UIs in Blade + PHP without writing JS — server renders, AJAX round-trips bring updated HTML fragments. Great for teams that don't want a separate SPA stack.
Tradeoff: network chatty (every interaction = round-trip), so it's not ideal for highly interactive UIs. Vue/React keep state client-side, faster interactions, but need a separate build/deploy pipeline and API layer.
Flux in this context is Livewire's official component library (different from React's Flux pattern) — worth clarifying with the candidate.
§ 03

Vue.js — 6-year claim, expect expert level

6 questions · the deepest probe
Q 3.1EASY
In Vue 3 Composition API, what is the difference between ref() and reactive()?
Expected Answer
ref() wraps any value (primitive or object) in a reactive container — access via .value in JS, but auto-unwrapped in templates.
reactive() uses a Proxy and only works on objects/arrays — no .value needed but you lose reactivity on destructure.
Practical rule: ref for primitives and when you need to reassign whole objects; reactive for stable object shapes. Bonus: mention toRefs() for safe destructuring.
Q 3.2MEDIUM
Six years of Vue means the candidate lived through Vue 2 → Vue 3. Name three breaking changes that required code rewrites during migration.
Expected Answer
Strong candidates name several from:
  • new Vue() → createApp() — multiple roots, no more global state pollution.
  • Filters removed (use methods or computed instead).
  • Functional components syntax change — no more functional: true.
  • v-model now uses modelValue + update:modelValue; v-model.sync gone.
  • Event bus pattern dropped ($on, $off removed) — use external libs or provide/inject.
  • Reactivity rewritten with Proxy — can now react to property add/delete without Vue.set.
  • Fragments — multiple root nodes per template.
Red flag: If the candidate can only name one or two surface changes, the 6-year claim deserves further probing.
Q 3.3MEDIUM
What does this Vue 3 component log when the button is clicked?
<script setup>
import { reactive, watch } from 'vue';

const state = reactive({ count: 0, name: 'A' });

watch(() => state.count, (n, o) => {
  console.log(`count: ${o} → ${n}`);
});

function handler() {
  state.count++;
  state.name = 'B';
  state.count++;
}
</script>
A.count: 0 → 1, count: 1 → 2
B.count: 0 → 2 (single log, batched)
C.count: 0 → 1, count: 1 → 2, name change logged
D.Nothing — watch needs deep: true
Expected Answer
B. Vue batches reactive updates within a synchronous block — by default, watchers fire after the next tick with the final value. So even though count is incremented twice, the watcher fires once with old=0, new=2. Knowing this is the difference between someone who's used Vue and someone who's debugged Vue.
Bonus: ask about flush: 'sync' if you want immediate watch firing.
Q 3.4MEDIUM
Pinia vs. Vuex — what made Pinia the default, and what changes for a developer migrating from Vuex?
Expected Answer
  • No mutations — Pinia drops mutations entirely; actions can directly mutate state.
  • No modules nesting — flat stores instead of namespaced module trees.
  • Better TypeScript inference — first-class TS support without complex generics.
  • Composition API native — defineStore with setup-style stores is idiomatic.
  • Smaller bundle, devtools support.
Resume claims both Vuex and Pinia — candidate should be able to articulate why a team would migrate.
Q 3.5HARD
You're building a reusable DataTable component used across many pages. It needs: configurable columns, slot-based custom cell rendering, sort/filter state lifted to parent, and v-model on selected rows. Sketch the API (props, emits, slots) you'd expose.
Expected Answer
A senior Vue dev should sketch something like: 
props: {
  rows: Array,
  columns: Array,           // [{ key, label, sortable }]
  modelValue: Array,        // selected row ids
  sort: Object,             // { key, dir }
  filter: Object
}
emits: [
  'update:modelValue',
  'update:sort',
  'update:filter',
  'row-click'
]
slots: {
  'cell-{columnKey}': // scoped slot per column
  empty,
  loading
}
Strong signals: scoped slots per column, multiple v-model bindings (Vue 3), separation of state from presentation, accessibility considerations (aria-sort).
Q 3.6MEDIUM
What does <script setup> give you that the older Composition API setup() function doesn't?
Expected Answer
  • Compile-time syntactic sugar — top-level imports and bindings auto-exposed to template.
  • No need to return things from setup.
  • defineProps, defineEmits, defineExpose as compiler macros.
  • Better runtime performance (less object overhead per instance).
  • Better TypeScript inference for props.

React.js — 2-year claim, recent addition

5 questions · verify it's hands-on
Q 4.1EASY
Why does this component cause infinite re-renders?
function Profile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetchUser(userId).then(setUser);
  });

  return <div>{user?.name}</div>;
}
A.fetchUser is not memoized
B.Missing dependency array → effect runs every render → setUser → re-render → effect again
C.setUser needs to be wrapped in useCallback
D.It does not — React batches state updates
Expected Answer
B. Without a dependency array, useEffect runs after every render. Each run calls setUser, which triggers another render, which runs the effect again. Fix: useEffect(() => {...}, [userId]).
Q 4.2MEDIUM
When is useMemo actually worth it, and when is it premature optimization?
Expected Answer
Worth it when:
  • Computation is genuinely expensive (filtering/sorting large lists, complex transforms).
  • The memoized value is passed as a prop to a React.memo child — reference stability matters.
  • It's a dependency of another hook (useEffect, useMemo) and you need stable identity.
Premature when wrapping cheap computations — useMemo itself has overhead (closure + deps array comparison). For React 19's compiler, this becomes mostly unnecessary.
Q 4.3MEDIUM
Resume claims Redux Toolkit. Walk through what createSlice generates for you that vanilla Redux required manually.
Expected Answer
  • Action creators auto-generated from reducer function names.
  • Action types auto-generated as sliceName/reducerName.
  • Immer integration — write "mutating" reducers that produce immutable updates under the hood.
  • Returns { actions, reducer } — combineReducers handles the rest.
  • Bonus: mention createAsyncThunk, extraReducers, RTK Query for data fetching layer.
Q 4.4HARD
You wrap a child component in React.memo but it still re-renders whenever the parent re-renders. Give two reasons why memoization isn't working.
Expected Answer
  • Inline objects/arrays as props — { ...obj } or [a, b] creates a new reference every render, defeating shallow comparison.
  • Inline functions as props — onClick={() => ...} creates new function each render. Need useCallback.
  • Children prop is a new element every time (most overlooked).
  • Context consumption — if the child uses useContext for a changing context value, it re-renders regardless of memo.
Q 4.5EASY
You used React Testing Library (per resume). What's the philosophy behind it, and how is it different from Enzyme?
Expected Answer
RTL's philosophy: "The more your tests resemble the way your software is used, the more confidence they can give you." Query by accessible roles, labels, text — what users see, not internal component structure.
Enzyme tested implementation details (state, props, instance methods), making refactors painful. RTL forces you toward integration-style tests around the public surface. Good candidates mention screen.getByRole, userEvent over fireEvent, async findBy queries.

Databases & API design

5 questions · MySQL, Mongo, REST
Q 5.1MEDIUM
Given a MySQL orders table with columns (id, user_id, status, created_at, amount) and a query: SELECT * FROM orders WHERE user_id = ? AND status = 'paid' ORDER BY created_at DESC LIMIT 20. What's the optimal index?
A.Separate indexes on user_id, status, created_at
B.Composite index on (user_id, status, created_at)
C.Composite index on (created_at, user_id, status)
D.Full-text index on the table
Expected Answer
B. Leftmost-prefix rule: equality predicates first (user_id, status) then the range/order column (created_at). MySQL can use the composite index for the WHERE filter and avoid a filesort. Strong candidate also mentions covering index, EXPLAIN output, and that index cardinality matters.
Q 5.2MEDIUM
When would you choose MongoDB over PostgreSQL for a project, and what's the strongest argument against that choice?
Expected Answer
Choose Mongo when: schema is genuinely fluid, document-shaped data (embedded docs > joins), horizontal sharding from day one, write throughput dominates.
Argument against: PostgreSQL has JSONB, GIN indexes, partial indexes, MVCC, foreign keys, ACID across multi-row transactions. For most "we need flexibility" use-cases, PG + JSONB is the better default. Mongo's join story ($lookup) is weaker, and ad-hoc analytics are harder.
Senior signal: candidate doesn't reflexively pick one — frames the decision.
Q 5.3MEDIUM
Your REST API exposes POST /payments. A flaky client retries the request and you accidentally charge the user twice. How would you make this endpoint idempotent?
Expected Answer
Strong answer:
  • Client sends an Idempotency-Key header (UUID).
  • Server stores key + response in a table/Redis with TTL.
  • On a duplicate key within TTL: return the original response, do not re-execute.
  • Use a DB unique constraint on (idempotency_key, endpoint) to enforce at the storage layer.
  • Bonus: mention Stripe's pattern as the canonical implementation; race conditions handled via SELECT … FOR UPDATE or insert-then-catch-duplicate.
Q 5.4HARD
Offset pagination (LIMIT 20 OFFSET 100000) becomes brutally slow on big tables. What's the alternative, and what's its tradeoff?
Expected Answer
Cursor-based / keyset pagination: instead of OFFSET, use WHERE id < last_seen_id ORDER BY id DESC LIMIT 20. The DB seeks straight to the right index entry rather than scanning + discarding.
Tradeoff: no random page access — you can't jump to "page 47". Also harder if sort is by a non-unique column (need a tie-breaker, usually (sort_col, id)).
Good for infinite-scroll UIs, mobile feeds, log pagination.
Q 5.5EASY
What HTTP status code do you return when a resource exists but the user isn't allowed to access it?
A.401 Unauthorized
B.403 Forbidden
C.404 Not Found
D.Either 403 or 404 depending on policy — both can be valid
Expected Answer
Strictly B (403) per RFC. But D is a defensible senior answer — some APIs return 404 to avoid leaking resource existence to unauthorized users (auth probe protection). 401 is for unauthenticated, not unauthorized. Accept B or D with rationale.

DevOps, Testing & AI tools

5 questions · CI/CD, Docker, AI workflow
Q 6.1EASY
What does a multi-stage Dockerfile give you that a single-stage one doesn't?
Expected Answer
Smaller final images — you build in a fat image (with dev deps, compilers, npm cache) and copy only the artifact into a minimal runtime image. Benefits:
  • Smaller attack surface (no shell, no build tools in production).
  • Faster pulls and cold starts.
  • Same Dockerfile handles build + run; CI doesn't need a separate builder step.
  • Cache layers per stage.
Q 6.2MEDIUM
In a GitHub Actions workflow, what's the difference between on: push with a branch filter and on: pull_request?
Expected Answer
  • on: push fires on every commit to a matching branch (typically main for deploys, or feature branches).
  • on: pull_request fires when a PR is opened/updated/reopened — runs against the merge commit (not the branch head), so checks reflect what would land on main.
  • Common pattern: PR triggers run tests/lint; push to main triggers deploy.
  • Senior detail: pull_request from forks doesn't get secrets (security boundary); use pull_request_target with care if you need them.
Q 6.3MEDIUM
You're testing a function that hits an external API. Walk through three different mocking strategies in Jest and when to use each.
Expected Answer
  • jest.mock(moduleName) — auto-mock the whole module. Quick but coarse.
  • jest.spyOn(obj, 'method').mockResolvedValue(...) — surgical, preserves real behavior of the rest of the module.
  • MSW (Mock Service Worker) or nock — intercept at the network layer. Tests the real HTTP client code. Best for integration tests.
When: unit-test logic → spyOn; module needs whole-replacement → jest.mock; testing HTTP error handling, retries, headers → MSW.
Q 6.4EASY
Resume says you've been using ChatGPT, GitHub Copilot, and Gemini for 2 years. Walk me through a recent example — what task, what you asked, and where the AI was wrong or unhelpful.
Expected Answer
What you're looking for: concrete specifics. A genuine AI-tool user will name something like "Copilot kept generating outdated Laravel syntax for queues" or "Gemini hallucinated a Vue API that didn't exist" or "Used ChatGPT to scaffold tests and had to fix the matcher names."
Red flag: vague answers ("it speeds up boilerplate", "writes code for me") without examples. Means the candidate uses AI superficially.
Bonus: ask about prompt patterns, when they don't use AI, and how they verify output.
Q 6.5MEDIUM
Resume says "deployed and managed cloud applications on AWS EC2." Walk through how you'd zero-downtime deploy a Node + Laravel app on EC2. (No Kubernetes — just EC2.)
Expected Answer
Acceptable answers vary; senior signal is mentioning at least one of:
  • Blue/green: two EC2 fleets behind an ALB; deploy to green, swap target group.
  • Rolling: ASG with health checks; deregister, drain, replace one at a time.
  • For Laravel specifically: php artisan down --refresh=15, or symlink-swap pattern (Envoyer / Deployer) — release directories, atomic symlink to current/.
  • For Node: PM2 graceful reload, or systemd socket activation.
  • DB migrations are the gotcha — backward-compatible migrations only during deploy.
Red flag: "I just SSH and git pull then restart" — not senior-grade.
